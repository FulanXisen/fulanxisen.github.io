超越错误码：调试模糊问题的实践指南

[FanYuxin Hayden Riddiford]

2025/04/24

周五下午四点左右，你正处于高效工作状态，准备结束一周的工作。突然，一切都乱了套：日志完全停止输出，没有任何错误线索；LED不再闪烁；就连你精心编写的调试命令行界面也对任何指令毫无响应。你不禁对着空气大喊：“我明明没做什么复杂改动啊！“固件彻底变砖，而你设置的所有安全机制都没能告诉你问题出在哪里。

这类问题一旦出现，往往令人极度沮丧和不知所措。它们会严重打击你对代码库的信心，让你觉得任何微小的改动都可能随时引发问题。在我的职业生涯中，我遇到过很多类似的情况，每一次都是关于调试的痛苦教训。

好消息是，我遇到的所有问题最终都有合理解释。而且，尽管我有时希望问题出在别处，但至今还没有遇到过太空中的亚原子粒子冰雹导致的系统错误。

本文将提供一套结构化的、分步式的方法，帮助你调试那些表面上看似难以捉摸的错误。

调试步骤

我设计这些步骤的目标是，以最小的工作量获取最多的错误信息。随着步骤的推进，所需的工作量会逐渐增加。理想情况下，在执行这些步骤的过程中，你能获得足够的信息来定位并修复问题。在步骤结束后，我会分享几个来自个人经历的案例，展示如何应用这些步骤进行调试。

虽然本文描述的一些问题涉及硬件因素，但需要说明的是，本指南主要针对"软件引发"的错误。因此，以下问题（尽管同样需要修复）不在本文讨论范围内：

  • 电路板设计不良或错误
  • ESD损坏或IC一般性损坏/故障
  • 掉电或电源供应不足

步骤1：分析变更内容

这一步看似简单到可笑，但我自己也经常忘记。除非你的固件从未成功启动过，否则它肯定在某个时候正常工作过。比较代码库在"正常"和"故障"状态之间的差异，对于定位问题至关重要。你可能暂时不理解具体发生了什么，但没关系——即使知道问题可能出在哪个大致区域，也是有价值的信息。

找出能导致故障的最小变更。根据你上次提交的时间，这一步可能需要在上次正常工作的提交和当前故障状态之间进行增量变更测试。Interrupt上已有一篇关于使用git bisect的精彩文章，详细介绍了如何实现这一点。

一旦对问题可能的位置有了大致了解，就可以开始利用编译器和MCU内置的错误报告功能了。

步骤2：充分利用编译器错误报告功能

如果你使用供应商提供的自动生成构建系统的工具，可能会错过一些零成本的错误预防措施！编译器提供了多种编译标志和静态分析工具，可以大大简化你的工作。其中一个标志（-Wreturn-type）曾在我的一个案例中帮我立即发现问题。Embedded Artistry的Philip Johnston写了一篇关于编译标志的优秀博客，详细介绍了他使用的一些标志及其重要性。

步骤3：充分利用MCU错误报告功能

我使用过的几乎所有MCU都具备某种故障报告能力。我反复参考的Interrupt文章详细介绍了硬故障处理程序，特别是在Cortex-M平台上的实现。

我将在案例研究中描述的几乎所有问题，最终都表现为触发硬故障（从而进入硬故障处理程序）。供应商提供的HAL代码通常会定义默认的硬故障处理程序，如下所示：

void Default_Handler()
{
  while (1) {
  }
}
void HardFault_Handler()   __attribute__((weak, alias("Default_Handler")));

至少，将这些处理程序替换为能够记录故障信息的版本，可以立即为你指明问题方向：

void HardFault_Handler()   {
  log("发生硬故障！");
  asm volatile ("bkpt #0");
  while(1) {};
}

注意其中的ARM特定"断点"指令，用于在连接调试器时自动暂停执行。

不同MCU的故障报告详细程度各不相同，但你应尽可能获取有关当前故障的详细信息。例如，在基于Cortex-M7的MCU上，你可以了解故障是否由以下原因引起：

  • 无效的内存访问（有时会显示具体的错误地址）
  • 未对齐的加载或存储操作
  • 除零错误
  • 未定义的指令
  • MPU（内存保护单元）违规（例如，尝试写入标记为只读的区域）

有时你会很幸运，获得的信息能让你立即定位错误。否则，这些信息至少能为你提供一个大致的排查方向。例如，如果硬故障是由访问错误内存地址引起的，你就知道需要查找某种形式的内存"损坏"问题。虽然这可能涉及广泛的问题，但至少你知道了大致的排查方向。

在尝试了这些低工作量的选项后，让我们采取更主动的方法来隔离问题。

步骤4：通过日志和调试隔离问题范围

这一步的目标是将问题发生的范围缩小到尽可能小的代码片段。我不说"代码行"是因为有些问题（尤其是与堆栈/堆损坏相关的问题）不会每次都在同一位置发生。固件通常具有高度动态性，会响应来自外部世界的刺激。在这里，我们尝试消除尽可能多的变量。理想情况下，我们希望能够在每次测试中以相同的方式在同一位置重现故障。

实现这一目标的方法包括：

  • 在可疑故障区域前后添加日志语句，逐渐缩小范围，直到定位到问题代码
  • 连接调试器，尝试在"单步执行"代码时触发故障
  • 简化或模拟固件的某些方面，使运行更可重复（例如，将传感器固定到特定值或值集合）
  • 对于偶发性问题，在可疑代码周围设置尽可能详细的日志（系统状态、变量状态等），让固件运行一整夜以捕获问题。需要指出的是，Memfault的整个平台都致力于在已发布产品中捕获此类错误！对于云连接设备，你甚至不需要连接调试器，因为核心转储会自动上传以供后续分析。

如果你完成了这一步但仍然不太理解问题所在，不要气馁。例如，你可能知道在程序的特定函数调用处发生了错误的内存访问，但仍然不知道原因。你甚至可能在其他地方调用同一个函数，而它却不会触发故障！

一旦你大致定位了错误发生的位置，就可以开始深入研究更微妙、更细致的潜在根本原因。

步骤5：关注堆栈使用情况

我为自己制定了一条通用规则：当固件中出现丑陋、难以重现、完全不合逻辑的问题时，我会深呼吸并问自己：有没有可能是堆栈溢出？有几次，答案是肯定的。捕获和调试堆栈溢出本身就是一个值得单独写一篇文章的主题；不过，有几个简单的方法可以帮助你判断是否存在堆栈溢出问题。

你知道你的堆栈（或多个堆栈）位于何处吗？这听起来像是个愚蠢的问题，但堆栈在内存中的位置可能有多种不同的配置，尤其是在使用RTOS的情况下。这需要查看你的启动代码，了解初始堆栈指针的设置位置以及该地址的来源（硬编码、链接器符号等）。在执行上下文切换的RTOS中，系统中的每个线程/任务通常都有自己的堆栈，任何一个都可能溢出。

探测潜在堆栈溢出的最简单方法是增大堆栈（或多个堆栈）的大小，看看问题是否解决。更精确的方法包括在堆栈末尾附近使用"金丝雀值”，或利用MPU在堆栈溢出到禁止区域时生成故障。Embedded Artistry上的这篇文章详细介绍了"金丝雀方法”，而这篇文章则讨论了利用MPU等方法。另外需要注意的是，如果你使用RTOS，可能已经可以使用一些可选的堆栈溢出保护检查（通常以性能为代价）。例如，FreeRTOS提供堆栈保护功能。

在调试时，了解堆栈的"起始"和"结束"位置会很有帮助。如果你能在接近故障点的断点处暂停执行，可以读取堆栈指针，了解你离溢出状态有多近。

与堆栈类似，另一个可能的问题源是堆内存管理。

步骤6：关注堆内存管理

你的应用程序是否使用动态内存分配（如malloc()、calloc()、realloc()）？如果你回答"否"，你确定吗？一般来说，要使用任何内存分配函数（由标准库提供），必须定义一个"系统调用"函数_sbrk_r。为方便起见，许多"代码生成"类型的供应商工具会为你生成_sbrk_r的默认实现，以及定位堆栈和堆的链接器脚本和符号。C和C++标准库中还有许多函数（取决于实现）会在后台分配内存（如printf()、sprintf()、std::string、std::Vector等）。这两者的结合导致许多人（包括我自己曾经）在不知不觉中在程序中使用了动态内存分配。在开始新项目时，我喜欢首先使用如下占位符定义_sbrk_r，以确保任何意外的内存分配都能被发现：

char *_sbrk_r(int incr)
{
  log("内存分配被触发！");
  asm volatile ("bkpt #0");
  while(1) {};
  // 不可达
  return NULL;
}

动态内存分配本身并非坏事，但它会带来一类完全不同的错误。Oracle的这篇文章很好地总结了使用动态内存分配时可能遇到的一些错误。

当涉及堆内存时，还有一个在我们工作的裸机系统中比较常见的错误。通常，裸机系统上的堆和堆栈配置如下：

堆栈在地址空间中向下增长，而堆向上增长。就像两列火车在同一轨道上相向行驶，当其中一个变得足够大时，最终会开始破坏另一个。这产生的行为与堆栈溢出非常相似。幸运的是，由于任何堆扩展都是在代码中显式调用的，因此可以并且应该实现检查，在堆即将超出其边界时触发错误条件。一些RTOS甚至包含自己的堆管理方案，例如FreeRTOS和Zephyr。与堆栈检查类似，在问题代码执行前使用调试器停止程序并查看堆的状态（以及其中分配的变量），有时可以为潜在的内存损坏问题提供线索。

在检查了堆栈和堆之后，让我们更深入一层，更接近MCU本身。

步骤7：检查关键外设配置

现在我们进入了调试过程中不太愉快的部分，这可能被不客气地称为"病急乱投医"。到这个阶段，你已经排除了许多可能的错误原因，但仍然遇到硬故障、荒谬的内存访问等问题。现在，你需要仔细检查可能配置错误的"关键"外设。我将介绍几类属于这一类的外设以及它们的常见问题。

内存映射RAM/闪存

内存映射RAM/闪存是指可以在内存中映射外部RAM或闪存IC的MCU。许多读者可能熟悉的一个常见例子是Raspberry Pi Pico。该芯片（RP2040或新的RP2350）内部只有用于持久存储的不可变ROM内存，而是执行加载在外部闪存IC上的代码。“内存映射"意味着一旦正确配置，处理器可以将这些外部芯片上的内存位置视为位于正常地址空间中。正如我将在一个案例研究中演示的那样，错误配置这些IC之一可能导致非常奇怪的行为。这些设备的配置和启用内存映射的外设应该被仔细检查。

DMA

DMA（直接内存访问）是一种外设，其目的是在处理器执行其他操作时并行执行内存传输。使用DMA时应非常小心，因为如果使用不当，很容易损坏内存。C和C++不知道DMA在任何给定时间"正在做什么”，因此要格外小心，验证DMA是否当前正在操作某个内存区域。嵌入式工程师使用DMA时可能会做各种危险的事情，包括：

  • 指示DMA从即将超出作用域的数组复制内存，DMA会继续从不再有效的位置复制内存
  • 尝试在代码中修改DMA已经在积极写入的内存，两种操作都不会按预期成功
  • 将DMA与缓存内存一起使用（见下一个主题）

缓存

随着MCU变得越来越复杂，许多MCU开始配备指令缓存（ICache）和数据缓存（DCache）。虽然缓存可以显著加快程序执行速度，但它也带来了一些令人头痛的问题。简单来说，处理器的缓存会将常用数据的副本存储在一个特殊位置，访问速度比从RAM或闪存存储中获取数据快得多。通常，处理器会自动决定何时以及从缓存中获取什么（与实际内存位置相比）。当引入DMA等外设时，情况会变得复杂。DMA在操作时通常不会"通知"处理器缓存需要刷新，导致获取到陈旧/错误的数据。如果你怀疑缓存可能是问题的原因，请尝试关闭所有缓存。一些处理器甚至允许通过寄存器写入手动刷新缓存。因此，更精细的方法是在可疑操作之前添加缓存刷新调用。

MPU

一些MCU配备了"内存保护单元"（MPU）。MPU用于指定特定内存区域允许的操作。例如，一个区域可能是只读的并允许代码执行，而另一个区域可能是读写的但不允许代码执行。MPU的配置通常很复杂，配置错误可能导致意外的硬故障。例如，这位热心人士提供了一份非常详细的指南，介绍了为基于Cortex-M7的STM32芯片配置MPU的经验和挑战。

中断控制器

正确配置中断和中断处理程序涉及许多细节，鉴于本文的范围，我仅提供几点提醒：

  • 你是否有在中断和非中断上下文中都能访问的数据？如果有，请确保防止任何潜在的数据竞争。这篇文章可能会让一些人惊讶：volatile限定符不提供线程同步的手段。
  • 你的代码中是否有任何地方禁用/屏蔽中断？在这些"临界区"要格外小心，以避免错过中断并可能导致意外行为。
  • 如果使用RTOS，你是否使用了适当的中断/非中断API（如果适用）？例如，FreeRTOS有单独的"中断安全"和"非中断安全"API，必须正确使用以避免未定义行为。

步骤8：查阅MCU勘误表和应用笔记

将检查勘误表和应用笔记放在这么后面，是因为我们假设你至少已经浏览过MCU数据手册的勘误表部分。现在，我们需要重新查阅勘误表和应用笔记，希望能找到第一次阅读时错过的信息，并将奇怪的行为归因于硅片bug或已知问题。应用笔记可能非常有帮助，特别是对于上面列出的一些更复杂/关键的外设。以MPU的STM32示例为例，ST提供了一份专门关于DMA使用的应用笔记，其中甚至有专门的"提示和警告"部分！

步骤9：寻求帮助

当所有其他方法都失败时…寻求帮助并没有错。追踪并解决一个特别棘手的bug是非常有成就感的，但最终产品需要发货，工作日需要结束，人们需要休息。除了向团队成员寻求帮助或咨询Stack Overflow外，我发现以下方法也很有帮助：

  • 你的芯片的供应商特定论坛
  • 联系供应商并与现场应用工程师交谈，帮助调试问题
  • 查看使用你的芯片并可能以类似方式使用它的开源项目

案例1：创造性的堆栈破坏

我当时正在测试我将Segger的实时跟踪（RTT）移植到新兴的Zig编程语言的实现。我的移植（以及原始RTT源代码）有一个功能，允许你指定一个链接器段来放置RTT控制块。控制块是JLink调试器写入/读取的区域，用于实现与设备之间的数据传输。到那时，我已经验证了除手动控制块放置之外的所有其他用例。我感到非常得意，编写了最后一个测试，将RTT控制块放在一个新定义的链接器段中，位于RAM的末尾。突然，一切都崩溃了。几次日志输出后，硬故障开始出现。似乎我记录的内容和数量会影响程序的崩溃行为。

步骤5：关注堆栈使用情况

我很快注意到，在接近故障点时，变量突然开始出现错误值。等等…我的堆栈在哪里？哦，天哪！聪明的读者可能已经从上下文猜到了：我的堆栈是硬编码为从RAM末尾开始的。我对使用链接器符号放置堆栈起始地址的其他项目感到自满，以为为RTT控制块定义新的链接器段会防止意外覆盖堆栈。这个假设是错误的。我创造了一个怪物：进入日志函数时会将数据压入堆栈，但向控制块写入字节会主动破坏这些堆栈数据。因此，出现了奇怪的日志行为：几个字符成功传输后，接着就是硬故障。

案例2：未定义行为引发的问题

为简单起见，我有一个包含两个方法的类，如下所示：

class ThisShouldBeFun {

  public:
  ...

  int first_function() {
    // 大量复杂代码
  }

  void second_function() {
    // 有一些改变类内部状态的副作用
  }
};

我的代码出现了奇怪的行为，经过一些调试后，我意识到即使我调用的是first_function，second_function也不知何故被执行了！

步骤1：分析变更内容

我以前没有使用过first_function，尽管它已经被添加到类中一段时间了，所以它没有出现在最近的代码差异中。

步骤2：充分利用编译器错误报告功能

观察者可能已经发现了我的错误：我没有包含返回语句。根据C++标准，非void函数在没有返回值的情况下终止是未定义行为。在我的情况下，这种未定义行为似乎导致下一条指令是第二个函数的地址。我可以对C++类的内存布局以及这可能如何发生做一些解释，但我从未能够在Godbolt中重现这种情况。你只需要相信，在这个特定的情况下，未定义行为导致first_function通过调用second_function来终止。-Wreturn-type警告标志（包含在-Wall中）本可以为我节省很多麻烦。

案例3：串口数据丢失之谜

我参与了一个项目，其中两个板子通过UART串口相互通信。当时我正在加入这个项目，注意到一个现有问题：数据包（这些板子实现了Modbus协议）会失败CRC检查，需要重试命令。由于这个重试机制，通信仍然可以工作，尽管速度不如预期。最终，似乎突然之间，数据包中的CRC失败数量激增，使通信变得无法使用，需要立即关注。

步骤7：检查关键外设配置

于是，我们开始了对UART驱动程序代码的漫长而痛苦的排查。我们使用DMA在UART接收时自动将接收到的字节传输到内存缓冲区，然后在RTOS任务中处理该缓冲区，每当有字节可用时。无论我们如何仔细检查外设配置与数据手册，都无法摆脱导致CRC失败的坏字节。突破出现在我终于想起：这个MCU有缓存！关闭缓存立即解决了问题。不再有错误的字节，因此不再有CRC失败。这引导我们…

步骤8：查阅MCU勘误表和应用笔记

阅读关于我们MCU上缓存的应用笔记，我们很快意识到DMA完全不关心缓存的状态。一旦我们修复了存储UART字节的内存缓冲区的缓存管理，就可以重新打开缓存，同时消除神秘的坏字节。因为在实现缓存管理之前，缓存经常过时，所以接收到的字节是否正确几乎完全是随机的，取决于其他代码执行和缓存命中/未命中。因此，更改不相关的代码会改变缓存行为，从而改变我们获得坏字节的频率。

不断抛出故障的代码位于引擎的资产解压缩部分，该部分从.WAD文件（DOOM引擎的专有数据压缩格式，通常存储在软盘或CD-ROM上）解压缩资产，并通过我之前提到的区域分配器将它们存储在RAM中。令人沮丧的是，故障并不总是在同一个地方发生：有时能到达启动屏幕，有时几乎立即崩溃。当它确实到达启动屏幕时，引擎总是会发出许多关于损坏资产的警告，并且一些资产（角色纹理等）会丢失。使用调试器查看堆栈跟踪，很明显，有时在访问SDRAM中的内容时，会产生一个虚假地址（以0xFF开头）。

步骤7：检查关键外设配置

虽然我花了很多时间兴奋地让DOOM引擎的各个部分工作，但我从未想过要仔细检查一些初始MCU配置。毕竟，我有一个"已知良好"的生产版本配置，运行着实际的咖啡机固件。但经过许多天的努力，确信我在引擎实现中某个地方搞砸了，我找到了罪魁祸首：一个设置调用中的复制粘贴错误，导致SDRAM IC的一个引脚配置错误，在电气上处于浮动状态（仅在Doom项目中，实际生产代码配置正确）。碰巧，这个浮动的引脚控制接收到的数据的单字节屏蔽。你猜怎么着？它屏蔽的恰好是最高有效字节…所以，我基本上创建了一个"随机内存故障生成器"。当配置错误的引脚浮动得太高时，它会屏蔽MSB，将接收到的数据损坏为0xFFXXXXXX。这解释了引擎的奇怪行为：如果运气好，SDRAM读取错误只发生在非关键纹理上（并且不会生成MPU故障），游戏就能勉强到达启动屏幕。最终，在TK-02上运行Doom的努力是值得的：

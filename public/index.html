超越错误码：调试模糊问题的实践指南

周五下午四点左右，你正处于高效工作状态，准备结束一周的工作。突然，一切都乱了套：日志完全停止输出，没有任何错误线索；LED不再闪烁；就连你精心编写的调试命令行界面也对任何指令毫无响应。你不禁对着空气大喊：“我明明没做什么复杂改动啊！“固件彻底变砖，而你设置的所有安全机制都没能告诉你问题出在哪里。

这类问题一旦出现，往往令人极度沮丧和不知所措。它们会严重打击你对代码库的信心，让你觉得任何微小的改动都可能随时引发问题。在我的职业生涯中，我遇到过很多类似的情况，每一次都是关于调试的痛苦教训。

好消息是，我遇到的所有问题最终都有合理解释。而且，尽管我有时希望问题出在别处，但至今还没有遇到过太空中的亚原子粒子冰雹导致的系统错误。

本文将提供一套结构化的、分步式的方法，帮助你调试那些表面上看似难以捉摸的错误。

调试步骤

我设计这些步骤的目标是，以最小的工作量获取最多的错误信息。随着步骤的推进，所需的工作量会逐渐增加。理想情况下，在执行这些步骤的过程中，你能获得足够的信息来定位并修复问题。在步骤结束后，我会分享几个来自个人经历的案例，展示如何应用这些步骤进行调试。

虽然本文描述的一些问题涉及硬件因素，但需要说明的是，本指南主要针对"软件引发"的错误。因此，以下问题（尽管同样需要修复）不在本文讨论范围内：

  • 电路板设计不良或错误
  • ESD损坏或IC一般性损坏/故障
  • 掉电或电源供应不足

步骤1：分析变更内容

这一步看似简单到可笑，但我自己也经常忘记。除非你的固件从未成功启动过，否则它肯定在某个时候正常工作过。比较代码库在"正常"和"故障"状态之间的差异，对于定位问题至关重要。你可能暂时不理解具体发生了什么，但没关系——即使知道问题可能出在哪个大致区域，也是有价值的信息。

找出能导致故障的最小变更。根据你上次提交的时间，这一步可能需要在上次正常工作的提交和当前故障状态之间进行增量变更测试。Interrupt上已有一篇关于使用git bisect的精彩文章，详细介绍了如何实现这一点。

一旦对问题可能的位置有了大致了解，就可以开始利用编译器和MCU内置的错误报告功能了。

步骤2：充分利用编译器错误报告功能

如果你使用供应商提供的自动生成构建系统的工具，可能会错过一些零成本的错误预防措施！编译器提供了多种编译标志和静态分析工具，可以大大简化你的工作。其中一个标志（-Wreturn-type）曾在我的一个案例中帮我立即发现问题。Embedded Artistry的Philip Johnston写了一篇关于编译标志的优秀博客，详细介绍了他使用的一些标志及其重要性。

步骤3：充分利用MCU错误报告功能

我使用过的几乎所有MCU都具备某种故障报告能力。我反复参考的Interrupt文章详细介绍了硬故障处理程序，特别是在Cortex-M平台上的实现。

我将在案例研究中描述的几乎所有问题，最终都表现为触发硬故障（从而进入硬故障处理程序）。供应商提供的HAL代码通常会定义默认的硬故障处理程序，如下所示：

void Default_Handler()
{
  while (1) {
  }
}
void HardFault_Handler()   __attribute__((weak, alias("Default_Handler")));

至少，将这些处理程序替换为能够记录故障信息的版本，可以立即为你指明问题方向：

void HardFault_Handler()   {
  log("发生硬故障！");
  asm volatile ("bkpt #0");
  while(1) {};
}

注意其中的ARM特定"断点"指令，用于在连接调试器时自动暂停执行。

不同MCU的故障报告详细程度各不相同，但你应尽可能获取有关当前故障的详细信息。例如，在基于Cortex-M7的MCU上，你可以了解故障是否由以下原因引起：

  • 无效的内存访问（有时会显示具体的错误地址）
  • 未对齐的加载或存储操作
  • 除零错误
  • 未定义的指令
  • MPU（内存保护单元）违规（例如，尝试写入标记为只读的区域）

有时你会很幸运，获得的信息能让你立即定位错误。否则，这些信息至少能为你提供一个大致的排查方向。例如，如果硬故障是由访问错误内存地址引起的，你就知道需要查找某种形式的内存"损坏"问题。虽然这可能涉及广泛的问题，但至少你知道了大致的排查方向。

在尝试了这些低工作量的选项后，让我们采取更主动的方法来隔离问题。

步骤4：通过日志和调试隔离问题范围

这一步的目标是将问题发生的范围缩小到尽可能小的代码片段。我不说"代码行"是因为有些问题（尤其是与堆栈/堆损坏相关的问题）不会每次都在同一位置发生。固件通常具有高度动态性，会响应来自外部世界的刺激。在这里，我们尝试消除尽可能多的变量。理想情况下，我们希望能够在每次测试中以相同的方式在同一位置重现故障。
